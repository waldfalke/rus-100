# Онтологическое тестирование контрактов

Этот каталог содержит материалы и инструменты для онтологического тестирования контрактов CATMEPIM. Онтологическое тестирование дополняет техническое тестирование, фокусируясь на концептуальной целостности, семантической точности и онтологической полноте контрактов.

## Что такое онтологическое тестирование?

Онтологическое тестирование — это подход к анализу и тестированию контрактов, основанный на применении онтологических осей и их пересечений для выявления скрытых противоречий, неполноты и несоответствий на концептуальном уровне.

В отличие от технического тестирования, которое проверяет синтаксическую корректность, типовую совместимость и реализуемость контрактов, онтологическое тестирование фокусируется на:

- **Концептуальной целостности**: Согласованность понятий и концепций в контракте
- **Семантической точности**: Точность выражения смысла и намерений
- **Онтологической полноте**: Охват всех релевантных аспектов предметной области

## Методология

Мы используем трехступенчатую методологию онтологического анализа и тестирования:

1. **Базовый анализ по отдельным онтологическим осям** - применяется ко всем контрактам
2. **Избирательный анализ пересечений онтологических осей** - для важных контрактов и выявленных проблем
3. **Комплексный анализ всех пересечений** - для критически важных случаев

Подробное описание методологии можно найти в файле [ontological_testing_methodology.md](./ontological_testing_methodology.md).

## Основные онтологические оси

В нашем анализе мы используем следующие основные онтологические оси:

1. **Внутреннее/Внешнее (Internal/External)**
2. **Индивидуальное/Коллективное (Individual/Collective)**
3. **Статическое/Динамическое (Static/Dynamic)**
4. **Часть/Целое (Part/Whole)**
5. **Агент/Система (Agent/System)**

## Как проводить онтологическое тестирование

### 1. Выбор уровня анализа

В зависимости от критичности контракта и стадии разработки, выберите подходящий уровень анализа:

- **Уровень 1 (базовый)**: Для всех контрактов на ранних стадиях разработки
- **Уровень 2 (избирательный)**: Для важных контрактов или при выявлении проблем на уровне 1
- **Уровень 3 (комплексный)**: Для критически важных контрактов или сложных проблем

### 2. Подготовка запроса для ИИ

Используйте соответствующий шаблон промпта из файла [ontological_testing_methodology.md](./ontological_testing_methodology.md), заменив `[ТЕКСТ АРТЕФАКТА]` на текст анализируемого контракта.

### 3. Анализ результатов

Проанализируйте полученные результаты, обращая внимание на:
- Выявленные онтологические проблемы
- Рекомендации по улучшению
- Системные паттерны и "слепые зоны"

### 4. Документирование результатов

Сохраните результаты анализа в соответствующем подкаталоге `reports/`, используя следующий формат имени файла:
```
YYYY-MM-DD_[module]_[contract]_level[1-3].md
```

Также сохраните результаты в машиночитаемом формате в проектной онтологии:
```
python ontology/utils.py save_test_results [contract_id] [test_level] [results_file]
```

### 5. Интеграция с другими методами тестирования

Используйте результаты онтологического тестирования в сочетании с результатами технического тестирования для получения полной картины качества контрактов.

## Проектная онтология

Проектная онтология в каталоге `ontology/` предоставляет машиночитаемую базу знаний, которая хранит:

- Информацию о контрактах и их взаимосвязях
- Результаты всех онтологических тестов
- Выявленные проблемы и рекомендации
- Историю изменений контрактов и результатов тестирования

Это позволяет:
- Отслеживать эволюцию контрактов и проблем
- Анализировать связи между контрактами и их элементами
- Накапливать знания о системе и ее концептуальной структуре

Подробное описание проектной онтологии можно найти в файле [ontology/README.md](./ontology/README.md).
